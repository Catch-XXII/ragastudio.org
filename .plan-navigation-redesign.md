# Navigation Redesign: Left Sidebar → Apple-like Top Menu Bar

## Executive Summary

- **Objective**: Redesign site navigation from a fixed left sidebar (desktop) to a minimal Apple-like top navigation bar on desktop, while preserving mobile hamburger menu with slide-out navigation
- **Non-goals**: 
  - Change page layout or content structure
  - Modify existing category/post data or collections
  - Add new navigation items
  - Implement submenu/dropdown menus
- **Constraints**: 
  - Desktop only change (desktop top nav replaces sidebar; mobile unchanged)
  - Must maintain theme toggle (dark/light)
  - Must preserve mobile hamburger UX
  - Two-state responsive design (mobile <1024px, desktop ≥1024px)
  - No code commits; local testing only
- **Proposed approach**: 
  - Extract nav data to a shared component (`NavItems.astro`)
  - Create new `TopNav.astro` component with Apple-style design (minimal, clean, dark/light aware)
  - Update `BaseLayout.astro` to conditionally render top nav on desktop, remove left sidebar
  - Update `Sidebar.astro` for mobile-only functionality (hamburger + slide-out menu)
  - Adjust content wrapper margins to remove left offset
  - Add subtle hover states and active indicators
- **Estimated time**: 4-5 hours (phased for atomic commits)
- **Total phases**: 4
- **Total commits**: 5

## Proposed Design

### High-Level Design

**Two-part navigation system:**

1. **Desktop (≥1024px)**: Top navigation bar replacing sidebar
   - Spans full viewport width
   - Horizontal nav items (Home, Search, Categories)
   - Minimal padding/margin, clean aesthetic
   - Theme toggle button on right side
   - Active state underline or subtle background
   - Post counts optional (hidden by default, can toggle via class)
   - Fixed position or sticky (depends on layout preference)

2. **Mobile (<1024px)**: Preserved hamburger + slide-out menu
   - Hamburger button top-left fixed
   - Slide-out menu same width and behavior as current
   - Overlay click-to-close
   - No changes to mobile UX

**Shared nav data:**
- Extract nav item logic to a reusable TypeScript utility or component
- Both top nav and mobile menu consume same data source
- Active state detection remains in shared utility

**Theme toggle placement:**
- Option A: Move to top nav bar right side (Apple.com style)
- Option B: Keep in Header but make it work with top nav
- **Chosen**: Option A (top right of top nav bar, cleaner)

**Layout changes:**
- Remove `lg:ml-56` from content wrapper (no longer needed)
- Adjust Header styling to work with full-width layout
- Top nav positioned above Header (or integrated into Header)

### API / Interface Changes

**New `NavItems.astro` component:**
```astro
---
interface NavItem {
  href: string;
  label: string;
  icon: string;
  count?: number;
}

interface Props {
  items: NavItem[];
  currentPath: string;
  variant: 'desktop' | 'mobile'; // Different styling per variant
}
---
```

**New `navUtils.ts` utility:**
```typescript
export function buildNavItems(postCountByCategory: Record<string, number>): NavItem[]
export function isActive(href: string, path: string): boolean
export function formatCount(count: number): string
```

### Failure Modes & Edge Cases

| Case | Mitigation |
|------|-----------|
| Active state broken when switching themes | Active state detection is path-based, not theme-based; should unaffected |
| Navigation items overflow on narrow desktop (1024px-1200px) | Use responsive font sizes or hide post counts at xs lg breakpoint |
| Mobile menu doesn't close after navigation | Hamburger logic unaffected; keep existing close-on-navigate pattern (add if missing) |
| Theme toggle context switching | Keep theme toggle in single location (top nav right); clearly visible |
| Accessibility: keyboard navigation | Ensure all links focusable, semantic HTML, ARIA labels |
| SSR: nav items undefined at build | Always run getCollection in root component (already done) |

## Questions/Decisions Needed

1. **Post count badges in top nav**: Should post counts be displayed in desktop top nav?
   - **Recommendation**: Hide by default (`hidePostCounts={true}` in TopNav.astro)
   - **Rationale**: Apple.com top nav is minimal; counts are less critical for navigation
   - **Compromise**: Can be toggled via class/prop if desired later

2. **Top nav positioning**: Fixed (sticky top on scroll) or static?
   - **Recommendation**: Sticky (position-sticky, top-0, z-30)
   - **Rationale**: User can always access nav while scrolling; matches modern UX expectations
   - **Trade-off**: Takes up vertical space but improves UX

3. **Active state indicator**: Underline, background, or accent border?
   - **Recommendation**: Bottom border-2 in accent color (underline style)
   - **Rationale**: Clean, minimal, Apple.com style, doesn't need background
   - **Implementation**: `border-b-2 border-[var(--color-accent)]` on active item

4. **Theme toggle location**: Top nav right or elsewhere?
   - **Recommendation**: Top nav right side
   - **Rationale**: Cleaner than Header; user expects it in top bar
   - **Trade-off**: Remove from Header.astro to avoid duplication

5. **Header styling with top nav**: Keep Header centered in content, or adjust?
   - **Recommendation**: Keep Header centered in content (existing layout)
   - **Rationale**: No need to change; Header (ragastudio logo) is separate from navigation
   - **Alternative**: Could integrate logo into top nav left side
